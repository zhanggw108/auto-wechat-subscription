# 长文章写法推荐系统设计

## 背景

当前长文生成虽然能调用真实 LLM，但主文章容易滑向固定模板：问题、方法、实验、局限、配图建议。这个结构能保证论文解读完整，却让每篇文章都像同一套 checklist，也会把“配图建议”“发布前复核清单”“内部评分”等生产过程内容暴露给读者。

目标不是放弃事实完整性，而是把“后台校验项”和“前台文章结构”分开：后台仍要求覆盖研究问题、方法贡献、实验可信度、局限边界和作者判断；前台文章按论文特征选择不同叙事策略，让每篇文章有自己的读法。

## 目标

1. 每篇长文候选都有一个可见的推荐写法。
2. 系统默认自动选择写法，用户可以在生成长文前切换。
3. 主文章内部不再固定成几个模板标题。
4. 正文不出现内部制作说明、配图建议、发布前复核清单或后台评分。
5. 保留 3 个模块标题，继续支持前端分模块展示和刷新。

## 非目标

1. 第一版不做多轮写作规划。
2. 第一版不让 LLM 自由创造无限文章类型。
3. 第一版不重写次文章生成体系，只在主文章写法上收口。
4. 第一版不引入外部引用、作者机构、PDF 全文解析等新数据源。

## 文章写法类型

第一版固定 5 类写法。

### 评测复盘型

适合 benchmark、evaluation、SWE-bench、arena、leaderboard、agent eval 等论文。

叙事重点：

- 旧评测哪里失真。
- 新评测到底测了什么能力。
- 实验结果说明了哪些真实失败模式。
- 这个评测对后续模型或产品有什么约束。

### 机制拆解型

适合新模型、新框架、新系统、新训练或推理方法。

叙事重点：

- 论文解决的关键机制问题是什么。
- 核心设计为什么可能有效。
- 实验如何证明机制成立。
- 方法边界在哪里。

### 争议判断型

适合两条路线比较或容易引发判断分歧的论文，例如 RAG vs 长上下文、开源 vs 闭源、端侧 vs 云端、安全收益 vs 成本。

叙事重点：

- 争议双方各自成立的条件。
- 论文证据支持哪一边。
- 哪些结论不能被过度外推。
- 作者自己的判断是什么。

### 趋势切片型

适合多篇论文、热点和工具共同指向一个变化方向。

叙事重点：

- 这不是单篇论文孤立事件，而是一个方向变化。
- 多个信号共同说明了什么。
- 为什么现在值得关注。
- 后续该追踪哪些指标或论文。

### 应用转译型

适合 AI coding、agent、机器人、工具链、数据工程、工作流类论文。

叙事重点：

- 研究结论能转译成什么产品或工程启发。
- 哪些场景会受影响。
- 落地前还缺什么能力或证据。
- 实践者应该怎样读这篇论文。

## 推荐写法数据

每个 `TopicPackItem` 的 `score_detail` 中新增推荐写法字段，避免改动过多模型边界。

示例：

```json
{
  "recommended_narrative": {
    "type": "evaluation_review",
    "label": "评测复盘型",
    "reason": "标题和摘要命中 benchmark、evaluation、SWE-bench，核心价值是重新定义评测口径。",
    "alternatives": ["application_translation"]
  }
}
```

字段含义：

- `type`：后端稳定枚举值。
- `label`：前端中文展示。
- `reason`：一句中文推荐理由。
- `alternatives`：可选替代写法，最多 2 个。

## 推荐规则

第一版用代码规则选择，不让 LLM 决定类型。

规则按优先级匹配：

1. 标题、摘要、分类或方法摘要命中 `benchmark`、`evaluation`、`eval`、`SWE-bench`、`leaderboard`、`arena`：推荐评测复盘型。
2. 命中 `versus`、`vs`、`trade-off`、`comparison`、`long-context` 与 `RAG` 同时出现，或摘要明显比较两条路线：推荐争议判断型。
3. 命中 `framework`、`architecture`、`system`、`training`、`inference`、`optimization`、`distillation`：推荐机制拆解型。
4. 命中 `agent`、`coding`、`robot`、`tool use`、`workflow`、`deployment`：推荐应用转译型。
5. 如果多条热门信号共同指向一个方向，或无法明确归入以上类型：推荐趋势切片型。

替代写法从次高匹配规则里取，最多 2 个。

## 生成流程

1. 生成或刷新长文候选时，后端为每篇长文候选计算 `recommended_narrative`。
2. 前端在长文候选卡片展示推荐写法和推荐理由。
3. 用户选择某篇长文进入文章工坊时，默认使用推荐写法。
4. 文章工坊在“生成长文”按钮附近展示当前写法，允许用户切换。
5. 用户点击“生成长文”时，前端把选择的写法传给后端。
6. 后端把写法类型和对应叙事策略加入 LLM 输入。
7. LLM 输出主文章模块，后端按既有校验检查中文、可发布性和禁用内容。

## Prompt 约束

保留的固定结构：

- `# 今日 AI 论文与热点文章包`
- `## 主文章：长论文解读`
- `## 次文章 1：AI 热点`
- `## 次文章 2：arXiv 高热度文章速报`
- `## 来源清单`

主文章内部要求：

- 不固定小标题。
- 小标题只允许使用 `###`，不允许使用 `####`。
- 小标题必须服务于当前写法类型，不得直接使用固定 checklist。
- 必须覆盖研究问题、方法贡献、实验可信度、局限边界和作者判断，但这些不要求作为标题出现。
- 必须有清晰主线，避免面面俱到。

禁用内容：

- `配图建议`
- `image2`
- `封面图`
- `机制图`
- `发布前复核清单`
- `发布前必须`
- `当前学术价值参考分`
- `内部评分`
- `后台`

允许的谨慎表达：

- `从摘要和证据包看`
- `论文声称`
- `这个结论需要结合实验设置理解`
- `这里不能过度外推`

## 前端展示

长文候选卡片显示：

```text
推荐写法：评测复盘型
原因：标题和摘要命中 benchmark、evaluation、SWE-bench，核心价值是重新定义评测口径。
```

文章工坊显示：

```text
本篇写法：评测复盘型
[下拉选择其他写法]
[生成长文]
```

如果用户切换写法，状态文案显示：

```text
已切换为：应用转译型，重新生成后生效
```

## 校验

主文章生成后增加轻量校验：

1. 不包含禁用内容。
2. 不包含 `####`。
3. 至少包含 2 个 `###` 小标题。
4. 仍通过现有中文主文章校验。
5. 仍覆盖 `方法`、`实验`、`局限` 这类必要事实点，但不要求它们作为标题。

如果校验失败，直接返回失败原因，不生成本地兜底正文。

## 测试

后端测试：

1. 评测类论文推荐 `evaluation_review`。
2. 系统/框架类论文推荐 `mechanism_explainer`。
3. RAG vs long-context 类论文推荐 `controversy_judgement`。
4. Agent coding 类论文推荐 `application_translation`。
5. 无明显类型时推荐 `trend_slice`。
6. 主文章包含 `配图建议`、`image2`、`发布前复核清单` 或 `####` 时失败。
7. 生成长文时用户选择的写法会进入 LLM 输入。

前端测试：

1. 长文候选卡片展示推荐写法和原因。
2. 文章工坊默认使用推荐写法。
3. 用户切换写法后，生成请求包含所选写法。
4. 禁用内容失败时，前端展示后端错误，不显示成功。

## 迁移与兼容

旧 topic pack 没有 `recommended_narrative` 时，前端不报错，显示为空或默认 `趋势切片型`。

后端读取旧数据时，如果没有推荐写法，可按当前论文数据即时计算一个默认推荐，不需要批量迁移历史文件。

## 成功标准

1. 新生成的主文章不再出现 `配图建议`、`image2`、`发布前复核清单`、内部评分。
2. 新生成的主文章不再出现 `####`。
3. 不同论文能展示不同推荐写法。
4. 用户可以在生成前切换写法。
5. 主文章仍覆盖论文问题、方法、实验、局限和作者判断。
