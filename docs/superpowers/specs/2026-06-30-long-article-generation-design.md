# 长文章生成与图文预览优化设计

## 背景

当前长文章生成链路主要依赖结构化摘要字段和本地兜底模板。生成失败时会退回模板稿，虽然流程能跑通，但产物不像面向读者的文章，也无法支撑专业论文解读。机制图目前在正文之后独立生成，正文只写“配图建议”，图片不会插入关键段落，读者无法通过图理解论文机制。

本次优化目标是把长文章生成从“模板填充”改成“全文读论文、结构化写作、图文同步排版、前端可预览”的生产链路。

## 结论

采用薄集成方案，不接入整套外部论文助手系统。

- 论文全文解析优先复用成熟项目能力，候选为 `marker` 或 `docling`，用于 PDF 转 Markdown、抽取图片、表格和章节。
- 公众号预览参考 `doocs/md` 的 Markdown 到微信 HTML 体验，但不整套搬迁 Vue 应用。
- 正文风格增加独立降 AI 味阶段，可参考 `blader/humanizer`、`brandonwise/humanizer` 这类开源项目的写作痕迹检查规则，但目标是自然、克制、好读，不是绕过检测。
- 文章写作、图位规划、封面图、文中配图、image2 图片生成、稿件版本和前端工作台继续留在现有 FastAPI + React 应用内。

不使用本地兜底模板生成长文。LLM 失败时长文生成失败；LLM 成功但图片失败时保留正文，并在前端明确标记失败图片。

## 不做的事

- 不接入一整套独立论文助手 UI。
- 不把 `gpt_academic`、`paper-qa`、`Perfect_paper_analyzer` 作为主流程替代品。
- 不继续生成无来源、无图位、无读者结构的模板长文。
- 不使用“绕过 AI 检测”作为产品目标或质量指标。
- 不自动发布公众号。

## 用户流程

1. 用户在选题池选择一篇长文章候选。
2. 前端显示“待生成”状态。
3. 用户点击“生成长文”。
4. 后端下载论文 PDF 或 arXiv 源文。
5. 后端解析全文，抽取正文、章节、原图、表格和参考证据。
6. LLM 基于全文解析结果和证据包生成结构化文章方案。
7. LLM 对正文做一次降 AI 味改写，保留事实、引用和章节意图。
8. 后端按方案生成正文 Markdown，并为封面图、文中图位生成或引用图片。
9. 前端刷新文章工坊，展示完整长文预览，图片已经插入对应段落。
10. 若部分图片失败，正文仍保留，失败图位在预览和素材区显示错误。

## 生成流水线

### 1. 论文全文获取

长文章生成前必须取得全文。

输入来源按优先级：

1. `Paper.pdf_url`
2. 由 `arxiv_id` 推导出的 arXiv PDF URL
3. topic pack 的 `source_urls` 中可识别的论文 URL

如果无法下载全文，主文章生成失败，前端显示明确错误，不生成本地兜底正文。

### 2. 全文解析

新增一个轻量解析边界，建议命名为 `paper_parser.py`。

职责：

- 下载 PDF 到稿件目录或缓存目录。
- 调用 `marker` 或 `docling` 将 PDF 转成 Markdown。
- 保存解析产物：
  - `paper/fulltext.md`
  - `paper/figures/*.png`
  - `paper/tables/*.md`
  - `paper/parse.json`

解析结果至少包含：

- 标题
- 摘要
- 章节标题
- 正文片段
- 图片路径和相邻说明文字
- 表格路径和相邻说明文字
- 实验章节候选
- 局限章节候选

若解析失败，长文生成失败。

### 3. 写作计划

LLM 第一轮不直接写 Markdown，而是生成结构化 JSON。

建议字段：

```json
{
  "article_title": "中文标题",
  "reader_promise": "读者读完能理解什么",
  "thesis": "核心判断",
  "sections": [
    {
      "id": "problem",
      "heading": "为什么这篇论文值得读",
      "purpose": "解释研究问题",
      "source_refs": ["paper:section:introduction"],
      "must_include": ["问题定义", "已有方法缺口"]
    }
  ],
  "figure_slots": [
    {
      "id": "fig-method-overview",
      "after_section_id": "method",
      "kind": "generated_mechanism",
      "purpose": "解释方法流程",
      "source_refs": ["paper:figure:2", "paper:section:method"],
      "prompt_brief": "画出输入、核心模块、输出和验证路径",
      "caption": "方法机制图：从输入到实验验证"
    }
  ],
  "quality_check": {
    "uses_fulltext": true,
    "has_method_explanation": true,
    "has_experiment_caveat": true,
    "has_limitations": true
  }
}
```

后端只接受合法 JSON；缺少全文引用、方法、实验或局限时直接失败。

### 4. 正文生成

LLM 第二轮基于写作计划生成文章正文片段，不直接决定图片文件路径。

要求：

- 中文输出。
- 面向公众号读者阅读，而不是论文审稿提纲。
- 每个关键判断都能回到全文解析或证据包。
- 必须包含问题、方法、实验、局限、作者结论边界和我的判断。
- 不输出“配图建议”这种内部说明。
- 不复述论文摘要作为正文主体。

后端负责把正文片段和图位组合成最终 Markdown。

### 5. 降 AI 味改写

正文生成后增加独立风格改写阶段，输入为结构化正文片段、写作计划和引用约束，输出仍是结构化正文片段。

可复用方向：

- `brandonwise/humanizer`：优先参考。它有独立命令行和模式扫描思路，适合抽取“机械表达检查”和“修改建议”能力。地址：https://github.com/brandonwise/humanizer
- `blader/humanizer`：作为规则库参考。它是可移植写作 skill，适合借鉴“去掉 AI 写作痕迹”的编辑原则。地址：https://github.com/blader/humanizer
- `lynote-ai/humanize-text`：只作为调研参考。它的项目描述偏向绕过检测，不作为产品目标或默认依赖。地址：https://github.com/lynote-ai/humanize-text

改写规则：

- 保留所有事实、数字、论文名、实验边界和引用关系。
- 删除套话、总分总模板、空泛转折和“值得关注不是因为热”这类重复句式。
- 允许更像真人作者：有短句、有判断、有节奏，但不能口语到损害专业性。
- 不为了“人味”新增未经证据支持的比喻、故事或结论。
- 改写后再做一次事实引用检查，引用缺失或关键事实变形则失败。

### 6. 封面图和图位生成

封面图和文中配图必须属于同一套视觉系统。每篇文章生成一个统一的图片风格配置，建议字段为：

```json
{
  "style_id": "handdrawn-doodle-v1",
  "line_style": "clean hand-drawn ink line",
  "palette": ["深墨色", "纸白", "强调蓝", "少量暖色"],
  "label_style": "短中文标签",
  "composition_rule": "留白充足，主体清晰，避免小字和复杂背景"
}
```

封面图职责：

- 第一眼清晰，能看出文章讨论的论文主题或核心机制。
- 比文中机制图更有视觉冲击，但不脱离同一套手绘/涂鸦风格。
- 主体明确，构图简洁，适合公众号信息流缩略图。
- 不做抽象科技背景，不堆文字，不使用廉价渐变。
- 允许更醒目的标题感和中心视觉，但文字只保留必要短标签。

文中图位分两类。

`paper_original`：来自论文原图或表格。用于严谨引用，保留“论文原图”说明，不改写为事实结论。

`generated_mechanism`：由 image2 生成中文机制解释图。用于帮助读者理解关键机制。生成 prompt 必须来自写作计划和对应全文片段，不再使用固定的 planner/reviewer/executor 模板。

全篇生成图使用统一视觉风格：

- 简洁清晰，优先解释关系，不追求装饰感。
- 手绘风或涂鸦风，线条干净，构图留白充足。
- 同一篇文章的配色、线宽、箭头、图标颗粒度保持一致。
- 图中文字只保留必要中文短标签，避免小字堆满画面。
- 每张图只解释一个关键机制，不把整篇论文压进一张图。
- 封面图和文中机制图共享 `style_id`、配色、线条和图标语言；差异只体现在封面更抓眼，文中图更解释性。

最终 Markdown 示例：

```markdown
### 方法到底新在哪里

正文段落……

![方法机制图：从输入到实验验证](figures/fig-method-overview.png)

图中要看的不是装饰元素，而是三个关系：输入如何被拆解，核心模块如何协作，实验指标如何验证这个设计。
```

图片生成失败时，保留正文，并插入失败提示：

```markdown
> 机制图生成失败：image2 超时。正文已生成，请稍后重跑该图。
```

### 7. HTML 和前端预览

后端 Markdown 到 HTML 渲染必须支持图片。

前端文章工坊调整：

- 阅读模式展示完整主文章，不再只把模块正文放进 `pre`。
- 实时预览能渲染图片、标题、引用、列表和正文段落。
- 素材区继续列出所有图片资产，包括：
  - 类型
  - 路径
  - 插入位置
  - 生成状态
  - 错误信息
- 图片失败时，主文章仍可预览，并在对应位置显示失败说明。

## 数据模型

保留现有 `DraftAsset`，最小扩展字段：

- `status`: `ready | failed`
- `error`: 失败原因
- `slot_id`: 图位 ID
- `insert_after_section_id`: 插入位置
- `caption`: 图片说明
- `source_refs`: 原文引用来源
- `style_id`: 图片风格，例如 `handdrawn-doodle-v1`
封面资产也使用同一个 `style_id`，并标记 `kind=cover`。

新增解析产物不必先进入数据库，可先以稿件目录文件落盘，路径挂在 `Draft` 或 `evidence.json` 中。

## 错误处理

| 场景 | 行为 |
| --- | --- |
| PDF 下载失败 | 长文生成失败，不写正文 |
| PDF 解析失败 | 长文生成失败，不写正文 |
| LLM 写作计划失败 | 长文生成失败，不写正文 |
| LLM 正文失败 | 长文生成失败，不写正文 |
| 降 AI 味改写失败 | 保留原始正文片段，但标记需要人工复核 |
| 降 AI 味改写导致引用缺失 | 长文生成失败，不写正文 |
| 原图抽取失败 | 可继续生成正文，标记原图不可用 |
| image2 生成失败 | 保留正文，图位标记失败 |
| HTML 渲染失败 | 保留 Markdown，返回渲染错误 |

## 可复用项目判断

### doocs/md

适合参考公众号 Markdown 编辑和预览体验。本仓库前端是 React，不建议直接引入整套 Vue 应用。优先复用设计思想和必要的渲染规则。

### marker / docling

适合承担 PDF 到 Markdown、图片和表格抽取。实现时先做一层适配层，避免业务代码绑定具体库。若第一个库效果不好，可以替换适配层内部实现。

### paper-qa / gpt_academic / 其他论文助手

不作为主链路。它们解决的是论文问答、总结或通用学术助手，不直接解决中文公众号长文、图文插入、image2 资产和当前文章工坊预览。

### humanizer 类项目

`brandonwise/humanizer`、`blader/humanizer`、`lynote-ai/humanize-text` 等项目可作为降 AI 味规则参考。产品只吸收“识别机械句式、空泛转折、模板表达、过度概括”的方法，不接入“绕过检测”目标。实现优先级是先用规则和提示词约束，只有当本地规则检查不够稳定时，才考虑引入命令行工具。

## 测试边界

后端最小测试：

- PDF 解析适配层成功时生成 `fulltext.md` 和图片清单。
- PDF 解析失败时主文章生成失败。
- LLM 返回非法 JSON 时主文章生成失败。
- 降 AI 味阶段不能改变事实、数字和引用关系。
- LLM 成功、image2 失败时正文保留，资产状态为 `failed`。
- Markdown 渲染包含图片标签。
- 同一篇文章内封面图和文中机制图使用同一个 `style_id`。

前端最小测试：

- 主文章预览不再使用纯 `pre`。
- Markdown 图片能渲染为 `<img>`。
- 图片失败信息出现在正文对应位置。
- 素材区展示图片状态和错误。

## 分阶段落地

第一阶段只做主文章链路：

1. 去掉长文本地兜底模板。
2. 加全文下载和解析适配层。
3. 改 LLM 输出为写作计划 JSON 和正文。
4. 增加降 AI 味改写和事实引用检查。
5. 增加统一风格的封面图、图位插入和 image2 失败保留正文。
6. 前端支持图片预览。

第二阶段再考虑：

- 更完整的 doocs/md 样式兼容。
- 更多图位类型，例如对比图、实验表格图、结论边界图。
- 论文 LaTeX 源码解析。
- 图位重排和人工替换图片。
